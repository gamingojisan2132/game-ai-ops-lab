# 意思決定ログ

仕様判断・技術判断を記録するファイルです。

後続のAIエージェントが「なぜその判断になったのか」を追えるように、決定内容、理由、影響、見直し条件を短く残してください。

## 記録形式

```md
## YYYY-MM-DD: 決定名

- 決定:
- 理由:
- 影響:
- 採用しなかった案:
- 見直し条件:
```

## 2026-05-09: Unityではなく軽量なWebゲーム環境を優先する

- 決定: 初期段階ではUnityではなく、軽量なWebゲーム環境を優先する。
- 理由: このリポジトリの目的は、AIエージェントが小さなタスクを継続的に実装し、確認し、記録しやすい運用を作ることです。Webゲーム環境はファイル構成、差分確認、ブラウザ確認、CI確認を小さく始めやすいです。
- 影響: Unity固有のプロジェクト構造、アセット管理、エディタ依存の作業は当面扱わない。
- 採用しなかった案: Unityで最初からプロジェクトを作成する。
- 見直し条件: 3D表現、物理、エディタ上の制作フローが主要価値になったとき。

## 2026-05-09: Phaser / three.js / 素のWeb実装を候補にする

- 決定: 初期のWebゲーム技術候補は Phaser、three.js、素のWeb実装にする。
- 理由: Phaserは2Dゲームに向き、three.jsは3D表現に向き、素のWeb実装は依存を増やさずに小さなUIやロジックを試せます。
- 影響: 当面はこれら以外のゲームエンジンや大型フレームワークを不用意に追加しない。
- 採用しなかった案: 最初から多数の候補を比較する。
- 見直し条件: 最初のプロトタイプで明確に不足が出たとき。

## 2026-05-09: UI中心の2DゲームはPhaserを第一候補にする

- 決定: 現時点では、UI中心の2DゲームはPhaserを第一候補にする。
- 理由: スマホ縦画面の小ゲーム、入力、シーン管理、スプライト、簡単な物理やアニメーションを小さく実装しやすいため。
- 影響: `docs/GAME_IDEAS.md` の初期案は、Phaserで1〜3日程度で試作できるものを優先する。
- 採用しなかった案: three.jsを2D UIゲームの第一候補にする。
- 見直し条件: 3D表現が遊びの核になる、またはPhaserが過剰になるほど単純な実装だけで足りると分かったとき。

## 2026-05-09: AIエージェント運用を優先し、ドキュメント駆動で進める

- 決定: ゲーム本体の作り込みより先に、AIエージェント運用を優先し、ドキュメント駆動で進める。
- 理由: 空リポジトリの段階でタスク、判断、作業ログ、レビュー観点を整えると、後続のAIエージェントが迷わず小さな作業を継続できます。
- 影響: 実装タスクを始める前に、`docs/TASKS.md`、`docs/DECISIONS.md`、`docs/WORKLOG.md`、`docs/REVIEW_GUIDE.md` を更新する運用を前提にする。
- 採用しなかった案: まずゲーム本体を作り、後から運用ルールを整える。
- 見直し条件: 開発運用が十分安定し、プロトタイプ実装の比重を上げる段階になったとき。
