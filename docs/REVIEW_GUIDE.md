# レビューガイド

AIの作業結果を人間が確認するための観点です。

レビューでは、成功したように見える結果だけで判断せず、実行結果、画面表示、ログ、ドキュメント更新、回避策の有無を分けて確認してください。

## 必須確認

- buildは通ったか。
- 実画面は表示されたか。
- ブラウザコンソールエラーはないか。
- 回避策を使っていないか。
- 回避策を使った場合、通常の直し方とデメリットが記録されているか。
- 依存関係を不必要に増やしていないか。
- `README.md` と実態が一致しているか。
- 次のタスクが小さく切られているか。

## 変更範囲の確認

- 変更目的が1つに絞られているか。
- 関係ない整形やリファクタが混ざっていないか。
- 既存ファイルの大規模削除が行われていないか。
- 認証情報、秘密鍵、トークンが作成、保存、変更されていないか。
- 原因不明のまま依存関係を大量追加していないか。
- エラーを隠すためにテストやビルドを無効化していないか。

## ドキュメント確認

- 人間が読む文章は日本語で書かれているか。
- `docs/TASKS.md` に作業結果と次の小タスクが残っているか。
- `docs/WORKLOG.md` に実行した作業と確認結果が残っているか。
- 仕様判断や技術判断があれば `docs/DECISIONS.md` に残っているか。
- 回避策を使った場合、理由、通常の直し方、デメリット、将来戻すべきか、環境問題かプロジェクト問題かが記録されているか。

## フロントエンド確認

- HTTP 200だけで成功扱いにしていないか。
- JavaScriptとCSSが読み込まれているか。
- 実画面が意図通り表示されているか。
- ブラウザコンソールにエラーがないか。
- 主要操作を実際に試したか。
- スマホ縦画面でUIやテキストが重なっていないか。
- 実画面確認できない場合、「未確認」と明記されているか。

## ゲーム確認

- 最小ゲームループが成立しているか。
- プレイヤーの入力に反応があるか。
- 成功、失敗、リトライの流れが分かるか。
- スコア、HP、時間などの状態が画面上で確認できるか。
- ランダム要素がある場合、極端な結果で破綻しないか。
- 1つの変更でゲーム全体を作り込みすぎていないか。

## レビュー結果の記録例

```md
## YYYY-MM-DD: レビュー対象

### 確認したこと

### 問題

### 回避策の有無

### 残るリスク

### 次の対応
```
